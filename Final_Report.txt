1. Problem Statement

Households in Allegheny County who need public benefits for food assistance, health coverage, or utility help often struggle to understand whether they qualify before investing time in a full application. Eligibility rules are long, income thresholds change by household size, and multiple programs have overlapping but distinct criteria. A non-expert reading the PA DPW policy manuals cannot easily self-screen.

Our system provides a plain-language prescreening tool that takes a household narrative, extracts relevant facts, retrieves the applicable policy sections from real government documents, and returns a prioritized list of programs with evidence-backed explanations. The goal is not to replace caseworkers. It is to give households a grounded, honest first answer and a concrete next step.

2. Architecture and Design Choices

The system uses a three-agent pipeline:

User Input
  ->
Intake Agent - extracts structured profile; triggers clarification if incomplete
  ->
Eligibility Agent - RAG retrieval per program, LLM scoring, and rule cross-check
  ->
Explanation Agent - plain-language output, checklist, and caveats
  ->
Streamlit UI Output

2.1 Orchestration Decision

We use custom Python orchestration rather than LangChain, LangGraph, or another agent framework.

This choice matches the current implementation and evaluation setup. The project's final workflow is deterministic and evaluation-driven, and the runtime follows a bounded sequence of stages with explicit handoffs: intake, eligibility and prioritization, and explanation. In the current codebase, the top-level orchestration is implemented directly in src/rgnavigator/pipeline.py, while app.py controls when the workflow starts, when clarification turns continue, and when the system advances to the full pipeline.

Custom Python orchestration made the handoffs, stopping conditions, uncertainty flags, and Streamlit session-state behavior easy to inspect in both the code and the evaluation artifacts. This was especially useful for a course submission where the system needed to remain understandable, auditable, and easy to debug. It also reduced late-stage integration risk, kept the implementation aligned with the existing tests and evaluation artifacts, and fit the team's stronger familiarity with Python.

This should be understood as a structured agentic workflow with explicit handoffs and bounded decision logic, not as an open-ended autonomous agent. The current system does not implement LangChain, LangGraph, or another orchestration framework, and it is not described here as a production-deployed workflow manager.

Future work:
- If the system expands toward richer tool routing, tracing, deployed observability, external integrations, or broader workflow management, we may later migrate or retrofit the orchestration into LangChain, LangGraph, or a similar framework.

2.2 Intake Agent

The intake agent takes a free-text household description plus optional structured override fields. It uses gpt-4o-mini when an API key is available to extract a normalized UserIntake profile, including household size, income, employment, insurance, and special signals such as pregnancy or utility crisis. If required fields are missing or contradictory, the agent generates targeted clarification questions and triggers a multi-turn conversation before the rest of the pipeline is run.

Fallback: regex-based extraction when no API key is present.

2.3 Eligibility Agent

For each program in scope, including SNAP, Medicaid/CHIP, LIHEAP, and any user-uploaded programs:

1. RAG retrieval - generates a program-specific query from the intake profile and retrieves the top-k most relevant policy chunks from the local vector index using cosine similarity against OpenAI text-embedding-3-small embeddings.
2. LLM scoring - gpt-4o-mini scores eligibility as strong_match, possible_match, or no_clear_match using the retrieved chunks as grounding context, and produces a rationale with policy section citations.
3. Rule cross-check - a deterministic scoring function applies explicit thresholds from data/program_profiles.json, including income limits, household size, and special conditions.
4. Decision status - ready_for_explanation when intake is complete and no recommended program remains only a possible_match; ambiguous when intake is incomplete or any recommended program remains a possible_match.

Priority order is the order in which the navigator suggests a user review recommended programs first. It is a triage or review order based on likely applicability, program-profile fit, immediate hardship urgency, and uncertainty-aware screening signals. Missing or contradictory evidence should increase caveats or human follow-up, not create overconfident ranking. It is not an official statement of legal eligibility certainty, benefit amount, or application difficulty.

2.4 Explanation Agent

The explanation agent takes the ranked program matches and retrieved evidence. It uses gpt-4o-mini to produce:
- Plain-language eligibility summary per program
- Concrete next steps
- Itemized checklist of documents and actions per program
- Visible caveats, including uncertainty flags and the prescreening disclaimer

Fallback: template-rendered output when no API key is present.

2.5 Policy Retrieval Index

The system uses 2,191 chunks from PA DPW policy manuals for SNAP v8, Medicaid/CHIP, and LIHEAP, embedded with text-embedding-3-small and stored locally. New documents can be appended incrementally without rebuilding the full index. A local hashed bag-of-words fallback is available but less accurate.

3. Implementation or Build Summary

Key design choices

Per-program RAG queries
Each program generates its own targeted retrieval query rather than sharing a single query across all programs. This prevents Medicaid-specific policy text from contaminating SNAP or LIHEAP scoring.

Hybrid cross-check
Rather than trusting the LLM alone, every eligibility decision is independently scored by a deterministic rule engine. Missing or contradictory evidence is meant to surface as caveats, possible_match outcomes, or human-followup signals rather than as overconfident ranking. This limits false positives in a high-stakes context.

Incremental indexing
Early versions rebuilt the entire vector index on every document upload. Phase 3 replaced this with append_document_to_policy_index(), which embeds only the new document's chunks and appends them, making uploads fast even with a large existing corpus.

Multi-turn intake
The system does not fail silently when required fields are missing. It detects which fields are absent and generates specific follow-up questions, supporting up to 4 clarification rounds before proceeding with the best available intake state.

Phase 3 changes from Phase 2

- Retrieval: Per-program query generation, instead of a single shared query
- Indexing: Incremental append on upload, instead of full rebuild
- UI: Program selector added to the sidebar
- Scoring: Retrieval bonus double-count removed
- Sorting: Program sort key unified with priority order
- Retrieval: Medicaid terms moved to a Medicaid-only query layer
- Naming: LIHWAP and LIHEAP disambiguation fixed
- Testing: Cross-test contamination from uploaded files fixed

4. Evaluation Setup

4.1 Methodology

The evaluation set contains 10 cases covering standard success cases with complete intake and clear program need, multi-turn intake cases where income is withheld initially, edge cases including the pregnancy pathway and children-only coverage, and failure cases including missing income, contradictory data, and out-of-county input.

Each case was run through the live application and evaluated against expected behavior: whether intake completed correctly, whether the right programs were recommended, and whether the decision status accurately reflected system confidence. The evaluator treats eligibility correctness separately from priority-order limitations. Recommended-program correctness is primary, while priority order is checked directionally through limited acceptable top-priority sets rather than exact full-ranking matches.

Intake behavior:

5. Results and Failure Analysis

GOV-01 - Out-of-County Inputs Now Stop Before Normal Recommendations (AGENT_10)
Severity: mitigated

A Philadelphia household description is now flagged as outside the supported county scope, and the runtime stops before normal recommendation flow.

Root cause before the patch:
Intake scope detection existed, but the pipeline did not short-circuit after that warning.

Current behavior:
Intake marks the case insufficient_data, the final response is needs_human_followup, and the system withholds normal recommendations and normal priority_order. This case now passes the evaluator.

Remaining risk:
Unsupported geography phrased in ways the intake parser misses could still evade the guard.

GOV-02 - Contradictory Core Intake Is Suppressed Before Actionable Recommendations (AGENT_08)
Severity: mitigated

A user reported full-time employment with zero earned income this month. The runtime now preserves the contradiction, suppresses normal actionable recommendations, and surfaces needs_human_followup.

Root cause before the patch:
The pipeline could continue into recommendations even when core income and employment evidence conflicted.

Current behavior:
The evaluator now records this case as PASS because contradiction handling suppresses normal recommendations until the conflict is resolved.

Remaining risk:
Other contradiction patterns may still need explicit coverage in intake validation.

LIMIT-01 - Clear No-Match Guardrail Still Regresses on a High-Income Insured Adult (AGENT_02)
Severity: high

The high-income insured adult case should return a clean no-match result, but the current run still surfaces Medicaid/CHIP, LIHEAP, and SNAP as possible_match recommendations.

Root cause:
The current scoring and retrieval path still produces false-positive secondary match signals on this guardrail case.

Current behavior:
The evaluator still marks this case FAIL because SNAP and LIHEAP should not be recommended here.

Remaining risk:
This remains the clearest false-positive risk in the current best run.

LIMIT-02 - Pregnancy Pathway Case Still Misranks Priority and Misses Explanation Detail (AGENT_03)
Severity: medium

The pregnancy-specific health coverage case still fails because LIHEAP is ranked first, secondary programs are over-surfaced, and the explanation misses the term pregnancy.

Root cause:
The current ranking and explanation path still underweights the pregnancy-specific Medicaid pathway and over-surfaces secondary programs.

Current behavior:
The evaluator still marks the case FAIL because Medicaid/CHIP should be the primary review candidate in this scenario.

Remaining risk:
This is the clearest remaining priority-order limitation in the current best run.

LIMIT-03 - Graceful Degradation Is Safe but Still Misses Expected Detail (AGENT_06)
Severity: low

When the user declined to share income after two turns, the system correctly avoided false program matches and returned caveated fallback guidance. The remaining gap is that the evaluator still sees the case as under-specified.

Root cause:
Intake still stays at needs_clarification and omits num_adults from missing_fields, while the reviewer expectation for this case is insufficient_data plus a fuller missing-field list.

Current behavior:
The evaluator still marks the case FAIL even though the runtime remains conservative and suppresses normal recommendations.

Remaining risk:
The safe degradation path is better than before, but intake-state and missing-field detail remain inconsistent.

LIMIT-04 - Uninsured-Children Case Still Over-Prioritizes LIHEAP Ahead of Health Coverage (AGENT_07)
Severity: medium

The uninsured-children case still recommends LIHEAP, SNAP, and Medicaid/CHIP, but the evaluator rejects the ordering because LIHEAP is ranked first instead of Medicaid/CHIP or SNAP.

Root cause:
The current priority heuristic still overweights a secondary hardship signal relative to the health-coverage need in this scenario.

Current behavior:
The evaluator still marks this case FAIL because the review order does not match the intended triage emphasis for uninsured children.

Remaining risk:
This remains a reviewer-facing priority-order miss even though the program set itself is otherwise plausible.

6. Governance and Safety Reflection

This Phase 3 system is a course prototype for prescreening and next-step guidance only. It is not an official eligibility determination tool, application-submission system, legal-advice service, or case-management platform.

6.1. Data handled by the prototype

The intake flow handles sensitive household prescreening information, including free-text household descriptions, county and ZIP context, household composition, pregnancy or disability-related signals, income and employment details, insurance status, and hardship indicators such as food insecurity or utility-heating strain. The system also derives a normalized intake profile, missing-field and contradiction flags, program recommendations, caveats, and explanation text.

6.2. Session-scoped handling and retention

The live Streamlit app keeps active user input in session memory while the session is open. This includes the form fields, the working raw intake payload, clarification-turn chat history, the merged multi-turn household description, and the final NavigatorSession object containing intake, eligibility, explanation, and retrieved evidence. On a new submission, the app resets the conversation-specific session state for that run. There is no dedicated user-facing "Clear session" control in the current prototype; session-scoped state is expected to clear when the browser session is refreshed, restarted, or closed.

The checked-in runtime code does not intentionally persist live household intake profiles or raw user sessions to a database or dedicated log file. However, the UI includes a raw/debug Session JSON view for reviewer transparency, so any screenshots, copied output, or exported demo artifacts can expose household details and would need restriction or redaction before broader deployment.

6.3. External model/API handling

When API-backed mode is enabled, the prototype sends user-provided household narratives and structured intake details to OpenAI for intake extraction, clarification-question generation, eligibility scoring, and explanation generation. It also sends policy text and retrieval queries to the embedding model when building or updating the local policy index. This means the API-backed path does involve external processing of household prescreening content and structured derived context.

6.4. Uploaded policy documents and index governance

Uploaded policy documents should be treated as reference material, not applicant records. The ingestion path stores uploaded policy text locally and appends it into the persistent local retrieval index so it can be reused in later retrieval. Users should therefore avoid uploading documents that contain household-specific personal information. This upload and indexing path is intended for policy or program-reference text, not for applicant files.

6.5. Access assumptions for the course demo

The current demo assumes a controlled local or course-review setting rather than a public deployment. The checked-in evaluation artifacts and demo fixtures are synthetic cases rather than live applicant records, and that is the safest intended use for review, screenshots, and demonstrations.

6.6. User-facing safety and privacy guidance

The prototype already presents itself as prescreening only and surfaces uncertainty rather than claiming official authority. For course demos, reviewers should use synthetic or fictional household details. Because the app includes raw/debug views for transparency, those views should be treated as reviewer-only and not shown in a public-facing deployment without redaction or access restriction.

6.7. Current limitations

The prototype is not production-deployed and does not currently implement authentication, role-based access control, formal retention enforcement, consent management, deletion workflows, or administrative controls for removing uploaded policy materials from storage and the index. It also does not provide a dedicated in-app clear-session or clear-uploads control. In fallback or no-key mode, the system avoids OpenAI calls for LLM steps and can fall back to local parsing and lower-accuracy retrieval behavior, but this comes with weaker extraction and retrieval quality.

6.8. Future production requirements

A production version would need a visible privacy notice before intake begins, explicit third-party processing disclosure when API-backed mode is enabled, authentication and role-based access if hosted, formal retention and deletion controls, redaction or removal of raw/debug views, and clearer governance for uploaded policy documents and index versioning. It would also need a deliberate separation between synthetic evaluation artifacts and any live-user operational data.

For a concise repo-level audit summary, see docs/PRIVACY_SESSION_GOVERNANCE.md.

7. Lessons Learned and Future Improvements

Hybrid scoring is worth the complexity.
Running a deterministic rule engine alongside the LLM and surfacing uncertainty through possible_match, caveats, and human-followup signaling, rather than just trusting the LLM, caught several cases where the system would otherwise be too confident. The cross-check adds latency but meaningfully reduces false positives.

Per-program retrieval prevents cross-contamination.
When we used a single shared retrieval query, Medicaid-specific sections were being retrieved and used to score SNAP eligibility. Moving to program-specific queries, each generated from the intake profile and program context, significantly improved the relevance of cited policy sections.

Multi-turn intake works but needs better failure guidance.
The clarification chat correctly held the pipeline back when income was missing and successfully resolved the case after a second turn. The failure mode, when the user refuses to share income entirely, produces safe but limited output. Graceful degradation still needs a richer fallback path.

Intake-level validation gaps are high-risk.
The out-of-county case showed that a governance issue can sit outside retrieval and scoring entirely. The recent patch fixed the direct AGENT_10 failure by stopping normal recommendation flow, but it also highlighted how much of the remaining evaluation drift now lives in fallback extraction and state-setting rather than in ranking alone.

Index management needs attention at scale.
Early versions rebuilt the full 2,191-chunk vector index on every document upload. This was too slow for a live demo. The switch to incremental appending made uploads much faster, but it also means the index can accumulate stale or duplicate entries if documents are re-uploaded. Version tracking for the index is a future need.
